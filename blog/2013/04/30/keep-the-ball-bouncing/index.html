Keep the Ball Bouncing

Apr 30th, 2013

Wherein I write something very short before bed

So I was listening to the Shop Talk Show, as I frequently do while doing dishes late at night. I really depend on web dev podcasts to get me through the daily monotony of dishes, which there are a lot of in our house, because we prepare most of our meals at home. But I digress…

So I was listening to Shop Talk Show, and, as usual, got a nice little nugget of information about something that I use on pretty much every site: modernizr.

Modernizr has become so common, and is baked into my preferred WordPress starter theme, Bones. But Chris & Dave made a great point: modernizr does lots of “stuff” in the background, testing to see if the browser supports dozens and dozens of features. It can be up to ~40k (uncompressed), but beyond that it is injecting and removing things from the page and doing all sorts of other things in the milliseconds before the page is rendered.

They cautioned that it’s not always necessary to use it, and pointed out that you can build your own version of modernizr that tests only for the things you want it to. In fact the download page pretty much forces you to do this.

It got me thinking. How often do I actually use modernizr? So far I have only used it for the CSS classes it spits out on the <html> element, which indicate whether or not a crappy old browser supports something, allowing for fallbacks to be used in the CSS. For example:

    .thing{
        color: #fff;
        text-shadow: 1px 1px 1px rgba(0,0,0,.4);

        .no-borderradius &{
            color: #373737
        }
     }

And when I think about it, I rarely use modernizr. Usually only once or twice per project, sometimes not at all. Yet I haven’t really thought about not using it, or perhaps building my own stripped down version at the end of the project, only including the feature tests I need.

I’ll definitely be thinking about this going forward, though. How about you?
